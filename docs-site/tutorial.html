How It Works

Every component follows the same pattern: define, deploy, connect.

1

Define

Write a Kubernetes CRD manifest for your agent, MCP server, or model config. Agents get a system prompt and tool references. MCP servers get a URL and protocol. Skills get registered in AgentRegistry.

2

Deploy

Apply the manifests with kubectl apply. MCP tool servers run as standard Kubernetes Deployments. Agent CRDs are reconciled by the kagent controller. KMCP handles the MCP protocol routing.

3

Connect

Agents discover tools through RemoteMCPServer CRDs. Multi-agent systems reference other agents as tools using type: Agent. The Management UI provides a chat interface for testing.

Key Patterns

Architectural patterns demonstrated in this demo.

Pattern Example When to Use
Triage + Specialists Triage Agent routes to Customer Service, Mortgage, Compliance Multiple domains, single entry point, clean user experience
Multi-Agent Coordinator Operations Center delegates to Ops Monitor + Incident Manager + IT Support Cross-domain correlation, unified reporting, issue investigation
Tool-Per-Domain Separate MCP servers for customers, policies, transactions, status, incidents Clean separation of concerns, independent scaling, fine-grained access control
Model Tiering GPT-4o for complex reasoning, GPT-4o Mini for high-volume customer interactions Cost optimization — use the cheapest model that can handle each task
Internal vs External Compliance Agent is internal-only, Customer Service is customer-facing Different access levels, data sensitivity, regulatory requirements
Live Infrastructure Access K8s Agent with 18 real cluster tools via kagent-tool-server Operational agents that need to inspect or modify running systems
